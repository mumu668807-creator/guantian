# 项目起点

这个项目一开始不是为了做一个更快的起卦工具。

现实里的大衍筮法太难完整执行。五十根蓍草，先去一，其用四十九；每一爻三变，六爻十八变。分草、挂一、四四数之、归余、记录，任何一步都容易乱。真正执行一次，需要时间、耐心，也需要一只很稳的手。

电子化首先解决的是机械部分：数目不要错，余数不要漏，记录不要丢。但它不能替代人的那一下选择。大衍筮法里最重要的动作，是“分而为二”。所以系统可以负责右取一、四四数之、合爻、成卦，但分草处必须由用户自己点下去。

# 为什么放弃随机起卦

随机数很方便，也很干净。按一下按钮，卦就出来了。

但那样太像工具，太不像一次起筮。它把人的参与压缩成“请求结果”，而不是“进入过程”。大衍的价值不只在最后六爻，也在十八次分草里。每一次分开的地方，都带着当下的犹疑、偏向、节奏和手感。

所以这个项目不追求最快得到结果。它把机械步骤电子化，把关键选择还给人。结果不是系统替用户随机出来的，而是用户十八次分草选择留下的痕迹。

# 为什么从 3D 转向 2D

最早的方向是第一视角 3D：静室、木案、宽袖、手臂、蓍草、烛火。这个方向有沉浸感，也有野心。

但早期 3D 很容易露出粗糙。模型粗，光影粗，动作粗，反而让画面像一段未完成的游戏原型。中国风在这种粗糙里很容易变硬，变成表面元素，而不是气息。

后来转向 2D 平面观天。木案、双袖、蓍草、远山、水流、草地，都可以更克制地放在一张淡彩画面里。2D 不必证明空间真实，它只需要把注意力留在案上，留在草之间，留在用户点下去的那一瞬。

# 为什么开始重视沉浸感

电子化会带来便利，也会带走仪式。

如果所有步骤都瞬间完成，用户只会看见答案，不会经历起卦。大衍筮法本来就慢。慢不是缺点，它让人从日常速度里退出来，把问题放稳。

所以后来开始加入停顿、移动、分堆、归余、记录。UI 不能抢画面，说明不能太多，按钮不能太满。真正要留下的是节奏：先看见五十，再去一；再分，再取，再数，再余。画面不需要热闹，但不能死。它要有一点风，一点水声，一点手边的重量。

# 为什么“此卦照见”不是算命

“此卦照见”不替用户决定命运。

它也不应该说某事必成、必败、必分、必得。卦象更像一面冷一点的镜子：照出当前结构，照出变化发生在哪里，照出用户可能误判什么，也照出现实里还有哪些行动空间。

六十四卦、卦辞、爻辞和动爻规则提供的是文本依据；模型只是在这个依据上整理语言。它应该清醒、克制，贴近用户所问之事。它可以有一点东方感，但不能装神弄鬼。它说的不是“天命如此”，而是“你现在可能正在这样用力”。

# 为什么去掉 binary、本卦标签等开发者信息

binary、source、original、changed 这些信息对程序很重要，对用户不重要。

它们是数据库骨架，是测试时的扶手。真正进入画面时，用户不应该看见这些骨架。案边札记里只需要写清楚是哪一卦、哪一爻、原文是什么。内部取辞规则可以存在，但不必把字段名端到用户面前。

去掉这些信息，不是减少准确性，而是把准确性藏回系统里。用户面对的应该是整理好的卦象，不是一张暴露结构的表。

# 当前阶段

现在项目已经走过了几个关键转向。

真大衍数学已经建立：五十先去一，其用四十九；每爻三变，每卦十八变；右取一、四四数之、余数归除、六爻成卦，都按规则推进。手动分草已经成为主流程，用户每一次点击都会真实影响结果。

六十四卦资料库已经接入，爻位按《易经》标准由下往上。取辞规则也已经独立出来，不靠临时拼凑。AI 照见的架构已经预留，本地 provider 可以接 MiMo，也可以以后换成 DeepSeek 或其他兼容接口。前端不暴露密钥，真实模型通过本地代理进入。

但这些完成之后，真正的问题反而更清楚了：

这个项目有没有气。

# 接下来真正要做的事

接下来不该急着堆功能。

更重要的是音频、节奏、留白、札记感、慢，以及画面里的活气。鸟鸣和水声要轻，不要变成素材展示。分草要稳，不要像 UI 动画。记录要像案边写下来的东西，不要像数据库查询结果。按钮要少，文字要短，停顿要有意义。

后面的迭代，应该继续问同一个问题：

这一步是在让起筮更清楚，还是只是在让界面更满？

如果答案是后者，就先不做。
