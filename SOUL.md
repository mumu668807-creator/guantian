# SOUL

这个项目不是一个更快的起卦工具，而是一处可以慢下来完成大衍筮法的网页空间。

系统负责机械、易错、重复的部分：数策、挂一、四四数之、归余、成爻、成卦、取辞。用户必须保留真正参与的一下：每一变亲手选择分草处。结果应来自 18 次真实分草，而不是随机数，也不是按钮后瞬间给出的答案。

## 项目气质

- 慢、克制、安静，有一点案边手作感。
- 更像进入一段仪式，而不是打开一个效率工具。
- 画面重点在木案、蓍草、双手、留白、远山、水声和风，不在功能堆叠。
- AI 不是神谕，也不是算命先生；它只是依据卦辞、爻辞、动爻规则整理一份清醒的案边札记。
- 用户看到的是整理后的卦象与文字，不是程序字段、调试信息和数据库骨架。

## 审美原则

- 主线是 2D 平面观天：淡彩、水墨、低饱和、自然材质、轻微呼吸感。
- 页面应像一幅可交互的案上画，不像 SaaS 面板、后台系统或营销落地页。
- 按钮要少，文字要短，提示要像轻声提醒。
- 动画要稳，尤其分草、取一、数策、归余，不能像游戏特效。
- 音频如果接入，应轻到像环境：远鸟、风、溪流，不做素材展示。

## 禁止事项

- 不要改回随机起卦。
- 不要把“分而为二”自动化；这是用户必须参与的核心动作。
- 不要做仙侠手游风、红金边框、玄幻粒子、发光按钮、过度模板化中国风。
- 不要让 AI 说“必成、必败、天命如此”这类决定命运的话。
- 不要在前端暴露真实 LLM API key，不新增 `VITE_LLM_API_KEY`、`VITE_OPENAI_API_KEY` 之类变量。
- 不要把 Supabase 用户系统做成账号中心、等级、社交、付费权限系统。
- 不要为了“信息完整”把 binary、source、original、changed 等开发者字段直接暴露给用户。
- 不要一次性大范围重构；每次只做一个可回退的小任务。

## 用户体验原则

- 用户先放下问题，再进入过程，再得到卦象与札记。
- 每一步要让人知道“现在在做什么”，但不要解释到打断仪式。
- 等待不是 bug；合理的停顿是体验的一部分。
- 一天一卦是仪式节制，不是产品限制。
- 如果某个新功能只是让页面更满，而不是让起筮更清楚，就先不做。
- 开发调试能力可以存在，但正式体验里要藏起来。
